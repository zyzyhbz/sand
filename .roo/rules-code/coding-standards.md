# 编码规范文档 - 用户登录与数据隔离系统

## 1. 通用规则

1. 所有新增文件必须在文件头部添加 JSDoc 注释说明模块用途
2. 使用 `const` 和 `let` 声明变量，禁止使用 `var`
3. 异步操作统一使用 `async/await`，避免回调地狱
4. 所有错误必须被 try-catch 捕获并返回有意义的错误信息
5. 禁止在代码中硬编码密钥、密码等敏感信息，必须从环境变量读取

## 2. 数据库规则

6. 所有数据库操作必须通过 `userService.js` 封装，路由层禁止直接操作数据库
7. 数据库查询必须使用参数化查询，禁止字符串拼接SQL
8. 数据库连接实例应在 `database/init.js` 中单例管理
9. 表结构变更必须在 `database/init.js` 中集中管理，使用 IF NOT EXISTS 防止重复创建
10. 数据库文件路径通过环境变量 `DB_PATH` 配置，默认为 `./database/app.db`

## 3. 认证规则

11. JWT Token 生成时必须包含 `userId` 和 `username`，禁止包含敏感信息
12. JWT Secret 必须从 `process.env.JWT_SECRET` 读取，未配置时拒绝启动服务
13. Token 有效期：普通登录2小时，"记住我"7天，通过环境变量可配置
14. 密码哈希使用 bcryptjs，salt rounds 固定为10
15. 密码验证失败时返回统一错误信息"用户名或密码错误"，禁止提示具体是哪个错误
16. 用户名长度限制3-20位，只允许字母、数字和下划线
17. 密码最少6位，不限制最大长度

## 4. 中间件规则

18. 认证中间件必须正确区分公开路由和受保护路由
19. Token 验证失败统一返回 401 状态码和 JSON 格式错误信息
20. 中间件验证成功后必须将用户信息注入 `req.user`，包含 `id` 和 `username`
21. 静态资源文件不需要经过认证中间件
22. `/health` 健康检查端点不需要认证

## 5. 路由修改规则

23. 现有路由文件修改时，在路由挂载点统一添加认证中间件，不在每个路由函数内部重复验证
24. 所有文件上传操作必须将文件存储到对应用户子目录 `uploads/user_{userId}/`
25. 所有报告生成操作必须将报告存储到对应用户子目录 `reports/user_{userId}/`
26. 报告查询接口必须按 `user_id` 过滤，禁止返回其他用户的报告
27. 文件删除操作必须校验文件属于当前用户
28. 原有路由的函数签名和返回格式保持不变，仅在必要时添加 `req.user` 参数使用

## 6. 前端规则

29. Token 存储在 `localStorage`，key 固定为 `auth_token`
30. 所有 fetch 请求必须通过统一的请求封装函数，自动附加 Authorization header
31. 所有 fetch 响应必须检查 401 状态码，收到 401 时清除 Token 并跳转 `/login`
32. 登录页面加载时，若 Token 有效则自动跳转主页
33. 主页面加载时，若 Token 无效或不存在则自动跳转登录页
34. 登录/注册表单提交前必须进行前端校验，给出友好的错误提示
35. 登录/注册失败时显示后端返回的错误信息，不使用 alert，使用页面内联提示

## 7. 安全规则

36. 所有 API 响应中禁止返回密码哈希
37. 注册接口添加频率限制：同一IP每分钟最多5次注册请求
38. 登录接口添加频率限制：同一IP每分钟最多10次登录请求
39. 用户子目录路径必须通过 `path.join` 构建，禁止直接拼接用户输入
40. 文件路径操作前必须验证路径在用户目录范围内，防止路径遍历攻击

## 8. 文件组织规则

41. 新增的 `database/` 目录必须添加到 `.gitignore`
42. SQLite 数据库文件禁止纳入版本控制
43. 数据库初始化逻辑在服务启动时自动执行
44. 用户子目录在首次文件上传时按需创建，不在注册时预创建

## 9. 错误处理规则

45. 数据库连接失败时服务应拒绝启动并给出明确错误信息
46. JWT Secret 未配置时服务应拒绝启动
47. 所有路由的 catch 块中必须 `console.error` 记录完整错误信息
48. 返回给客户端的错误信息不应暴露系统内部细节（如SQL语句、堆栈跟踪）

## 10. 兼容性规则

49. 修改现有路由文件时，必须确保原有的请求/响应格式不被破坏
50. 新增数据库的 reports 表与文件系统中的 JSON 报告文件共存，不删除原有数据
51. 现有报告文件中无 `user_id` 字段的历史数据，统一归属到一个系统管理账户或标记为公开数据
52. `server.js` 修改后必须确保所有原有路由端点仍然可访问
